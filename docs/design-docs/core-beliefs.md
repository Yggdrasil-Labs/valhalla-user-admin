# 核心信条

## 1. 仓库是唯一的真相来源

智能体需要的一切都必须在仓库中。不依赖外部 Wiki、Slack 上下文或口头知识。如果没有版本化在这里，对智能体来说它就不存在。

## 2. 执行优于记录

当一条规则重要时，将其编码为 linter、结构测试或 CI 检查。仅靠文档会腐烂。当文档不够时，把规则变成代码。本项目通过 ESLint (@antfu/eslint-config)、Husky pre-commit 和 commitlint 实施代码质量规则。

## 3. 渐进式披露

智能体从一个小而稳定的入口（AGENTS.md）开始，按需导航到更深层的内容。永远不要一开始就把所有东西塞进上下文。

## 4. 约束边界，内部自由

定义严格的架构边界（分层依赖方向、组件 props/emits 契约、Store 作为唯一 API 消费者）。在这些边界内，智能体可以自由选择实现方式。

## 5. 显式优于隐式

不要有魔法。配置、依赖和副作用必须在函数签名或模块导入中可见。唯一例外是 unplugin-auto-import 和 unplugin-vue-components 提供的自动导入——但这些在 `src/types/*.d.ts` 中有完整的类型声明。

## 6. 偏好无聊的技术

选择可组合、API 稳定、在训练数据中有良好表现的依赖。Vue 3 + Vite + Naive UI + Pinia 是成熟且文档完善的组合。不追新框架，不引入实验性依赖。

## 7. 低成本纠错优于阻塞门禁

偏好快速迭代。用后续 PR 修复，而不是无限期阻塞合并。在智能体优先的系统中，纠错成本低，等待成本高。

## 8. 持续垃圾回收

定期运行清理以捕获漂移、过期文档和模式违规。小的持续修复胜过大的重写。技术债务是高息贷款——持续小额偿还。

## 9. 品味即代码

人类的偏好和审美标准应编码为 linter 和结构测试，而不是留作非正式的团队默契。当一个模式反复被违反时，把它变成自动化检查。

## 10. 深度优先分解

当智能体卡住时，答案不是"再努力一点"。正确做法是问："缺少什么能力？"然后先构建那个能力。将目标拆解为更小的构建模块，逐个解决。
